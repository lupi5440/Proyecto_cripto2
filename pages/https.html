HTTPS y Seguridad en la Transmisión

HTTPS cifra la comunicación entre el navegador y el servidor, protegiendo contra escuchas y manipulación de datos en tránsito.

HTTP vs HTTPS

❌ HTTP (Inseguro)

  • Datos transmitidos en texto plano
  • Vulnerable a sniffing (Wireshark, tcpdump)
  • Vulnerable a MITM (Man-in-the-Middle)
  • Sin verificación de identidad del servidor
  • Los ISP/routers pueden modificar contenido
GET /login HTTP/1.1
Host: ejemplo.com
Cookie: session=abc123
username=admin&password=secreto123

↑ Todo esto es visible en la red!

✅ HTTPS (Seguro)

  • Cifrado TLS 1.2/1.3
  • Protege contra sniffing
  • Protege contra MITM
  • Certificados verifican identidad
  • Integridad de datos garantizada
GET /login HTTP/1.1
Host: ejemplo.com
[Datos cifrados con TLS]
→ AES-256-GCM, SHA-384
→ Perfect Forward Secrecy

↑ Atacante solo ve datos cifrados

Componentes de HTTPS

  • TLS Handshake: Negociación de cifrado y autenticación
  • Certificados SSL/TLS: Verifican identidad del servidor (CA firmados)
  • Cifrado Simétrico: AES para cifrar datos (rápido)
  • Cifrado Asimétrico: RSA/ECDHE para intercambio de claves
  • HASHES: SHA-256/384 para integridad

Mejores Prácticas

✅ Usar TLS 1.2 o superior (deshabilitar SSLv3, TLS 1.0/1.1)

✅ HSTS (HTTP Strict Transport Security) - forzar HTTPS

✅ Certificate Pinning para apps móviles

✅ Certificados de CA confiables (Let's Encrypt es gratis)

✅ Renovar certificados antes de expirar

✅ Configurar ciphers seguros (ECDHE, AES-GCM)

Headers de Seguridad

Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src https:
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
